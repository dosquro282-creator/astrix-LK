## План модуля голоса/видео/шаринга экрана (WebRTC)

### Цели

- Голосовые и видео‑каналы с минимальными задержками.
- Шаринг экрана или окна приложения со звуком.
- E2EE поверх медиа‑потоков (сервер не видит содержимого).

### Архитектура

- Отдельный компонент MediaService (SFU — Selective Forwarding Unit).
- Сигналинг (создание/подключение к комнатам, обмен SDP/ICE) через существующий HTTP/WS сервера B.
- WebRTC‑соединения между клиентами и SFU:
  - Rust‑клиент использует `webrtc-rs` или обёртку над нативным WebRTC.
  - Сервер‑SFU ретранслирует зашифрованные RTP‑потоки другим участникам канала.

### E2EE для медиа

- Для каждого голосового/видео‑канала — отдельный ключ `K_media_channel` по аналогии с текстовыми каналами.
- На клиенте:
  - До шифрования SRTP применяется дополнительный слой шифрования (например, SFrame или insertable streams).
  - Ключи `K_media_channel` распределяются между участниками так же, как симметричные ключи текстовых каналов.
- На сервере:
  - SFU не имеет доступа к ключам `K_media_channel`, работает только с зашифрованными пакетами.

### Этапы реализации

1. Добавить сигналинг для звонков (создание/подключение к медиа‑каналу через API/WS).
2. Интегрировать WebRTC‑библиотеку на клиенте и в отдельном SFU‑сервисе.
3. Реализовать E2EE‑слой поверх RTP (SFrame/insertable streams).
4. Встроить управление медиа в UI:
   - подключение/отключение голоса и видео,
   - выбор источника (камера, окно приложения, экран),
   - индикация говорящих участников.
