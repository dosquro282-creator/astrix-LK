# План реализации UI Astrix (Discord-like)

> **Контекст для продолжения в новом диалоге:**  
> Проект **Astrix** — аналог Discord на Rust (чат, серверы, голос через LiveKit/Opus).  
> Клиент: `client/`, UI на **eframe 0.28 / egui 0.28**. Сейчас весь UI в одном файле `client/src/ui.rs` (~3000 строк): экраны Auth и Main, состояние в `State`/`MainState`/`VoiceState`, интеграция с `net` (API/WS), `voice`, `voice_livekit`.  
> Цель: сделать **полноценный Discord-like UI** — структура, колонки, кнопки, анимации, тема, модульная архитектура, моковые данные.  
> Этот файл — единый план и чеклист; по нему можно продолжать реализацию в новом чате без доп. контекста.
> После каждого выполненного этапа обновляй чеклист
> Перед каждым этапом запрашивай подтверждение на продолжение

---

## ТЕХНОЛОГИИ И ОГРАНИЧЕНИЯ

- **Rust**, **egui** (eframe), модульная архитектура, production-ready код.
- **Состояние:** чистая архитектура, минимум `Arc`/`Mutex` только где нужно.
- **Тема:** тёмная по умолчанию, цвета в отдельном модуле.
- **Цель:** максимально точная по структуре и расположению элементов копия Discord (не упрощённая).
- **Отзывчивость:** приложение **не должно зависать** при отсутствии ответа от сервера (загрузка каналов, участников, сообщений и т.д.) — см. раздел 8.
- **Синхронизация в реальном времени:** при любом изменении на сервере (добавление на сервер, создание/удаление/переименование каналов, изменение ников, аватарок, ролей и т.д.) это должно **сразу отображаться у всех клиентов** — см. раздел 9.

## 2. КНОПКИ И ФУНКЦИОНАЛ

- **Правило:** каждая кнопка либо привязана к существующей логике Astrix, либо при нажатии пишет в консоль:  
  `println!("Feature not implemented yet");`  
  Никаких «мёртвых» кнопок.
- **Чеклист привязок:**
  - Home/DMs, выбор сервера, выбор канала — переключение `AppState`/`CurrentView`.
  - Отправка сообщения, вложения — существующие `api`/`ws`.
  - Pinned messages — если есть API/данные, иначе заглушка.
  - Mute/Deafen — к `voice_engine_tx` / LiveKit.
  - Остальные (Threads, Notifications, Search, Inbox, Help, GIF, Emoji, Stickers, Create Channel, Explore, Add Server, Settings) — заглушки с логом до реализации.

## 3. АНИМАЦИИ (как в Discord)

Использовать только возможности egui:

- **Инструменты:** `ctx.animate_bool()`, `ctx.animate_value()`, интерполяция цветов (e.g. `lerp`), простые easing (e.g. linear, ease-in-out через множитель).
- **Эффекты:**
  - Hover: плавное изменение фона (animate_bool для hover, lerp цвета).
  - Smooth color transitions при смене темы/состояния.
  - Channel selection: «слайд» — вертикальная полоска слева от выбранного канала (плавное появление через animate_bool/lerp).
  - Серверная подсветка слева: вертикальная полоска у активного сервера (аналогично).
  - Member list: плавное появление/скрытие (opacity/width через animate_value).
  - Кнопки: лёгкое изменение масштаба или цвета при нажатии (response.hovered, response.clicked).
  - Speaking indicator: пульсация или цветовая подсветка по данным голоса (lerp по времени или по флагу speaking).

Без резких смен; все переходы через анимации egui.

## 5. АРХИТЕКТУРА И МОДУЛИ

### 5.1 Структура файлов

```
client/src/
  main.rs          — точка входа, создание App
  app.rs           — App (eframe App trait), хранение state, вызов панелей
  state.rs         — AppState, CurrentView, ChannelType, Message, User, Guild, все enum/struct состояния
  theme.rs         — Theme, все цвета
  guild_panel.rs   — левая колонка (серверы, Home, Add, Explore)
  channel_panel.rs — вторая колонка (название сервера, категории, каналы, профиль внизу)
  chat_panel.rs    — центральная область (заголовок, кнопки, чат, input)
  member_panel.rs  — правая колонка (участники, роли, speaking)
  bottom_panel.rs  — нижняя панель (аватар, ник, mute, deafen, settings)
  components/      — переиспользуемые виджеты
    mod.rs
    buttons.rs     — стилизованные кнопки с анимацией
    avatar.rs      — аватар круг
    (при необходимости — другие)
```

- **Разделение:** каждый модуль отвечает за свою зону; `app.rs` только собирает layout и передаёт state/theme.
- **Минимизация borrow-checker:** state по возможности передавать по ссылке или разбивать на подструктуры, которые передаются в панели по отдельности; `Arc<Mutex<>>` только для shared mutable state (например, если потребуется из фоновых задач).

### 5.2 Ключевые типы в state.rs

- **AppState** — корневое состояние: текущий экран (Auth / Main), выбранный сервер, канал, список сообщений, участники, и т.д. Можно опираться на текущие `State`, `MainState`, но переименовать и вынести в state.rs.
- **CurrentView** — что показываем: DMs, Guild(server_id), при необходимости Channel(channel_id) внутри Guild.
- **ChannelType** — Text, Voice (и при необходимости Category).
- **Message** — уже есть в `net`; при необходимости обёртка или re-export в state.
- **User** — текущий пользователь и участники; при необходимости обёртка над данными из `net` (Member и т.д.).
- **Guild** — сервер (id, name, icon, порядок каналов); можно alias к `Server` из net или обёртка.

Цель: чистый, масштабируемый стейт; минимум дублирования с `net`, при необходимости тонкий слой маппинга в `state.rs`.

## 6. МОКОВЫЕ ДАННЫЕ

- **Серверы:** 3 штуки (id, name, icon/initials).
- **Каналы:** 5+ на сервер (текст + голос), с категориями (например 2 категории по 2–3 канала).
- **Пользователи:** 10 (id, username, avatar/initials, роль при необходимости).
- **Сообщения:** 30 (разные авторы, каналы, даты).
- Переключение серверов и каналов должно быть полностью рабочим: смена `CurrentView`/выбранного канала, обновление списка сообщений в центральной области (из мока или из API, если уже загружаем с сервера).

Моки можно хранить в `state.rs` или в отдельном `mock.rs` и инициализировать при старте Main-экрана.

## 8. НЕБЛОКИРУЮЩАЯ ЗАГРУЗКА (ОТСУТСТВИЕ ОТВЕТА ОТ СЕРВЕРА)

Приложение не должно зависать, если сервер не отвечает при загрузке каналов, пользователей, сообщений и т.д.

### 8.1 Принципы

- **Никакого блокирующего ожидания в UI-потоке.** В `update()` не вызывать `block_on`, синхронные HTTP-запросы с неограниченным ожиданием или `.await` без таймаута. Все долгие операции — только в фоне (tokio-задачи, уже существующие WS/API-паттерны).
- **Таймауты:** для любых запросов к API (каналы, участники, сообщения, список серверов) задавать разумный timeout (например 10–15 с). По истечении — считать ошибкой и не блокировать UI.
- **Состояния загрузки в state:** для каждой сущности (каналы сервера, участники, сообщения канала) хранить явное состояние: `Idle` | `Loading` | `Loaded` | `Error(message)` или аналог. UI по этому состоянию показывает спиннер/скелетон, контент или сообщение об ошибке.
- **Результаты в state только из фона:** фоновые задачи по завершении кладут результат (или ошибку) в общее состояние (например через существующую очередь событий или `Arc<Mutex<State>>`) и запрашивают перерисовку (`ctx.request_repaint()`). UI только читает state и рисует; не ждёт завершения запроса в том же кадре.

### 8.2 Реализация

- **Запросы при смене сервера/канала:** при выборе сервера/канала ставить состояние в `Loading`, запускать tokio-задачу (или отправлять сообщение в существующий фоновый цикл), которая с таймаутом запрашивает каналы/участников/сообщения. По ответу или таймауту — обновить state и вызвать `request_repaint`. Если уже есть `ws_events` / канал команд — использовать их; не дублировать блокирующие вызовы в `update()`.
- **Повтор при ошибке:** при состоянии `Error` показывать в UI кнопку «Повторить»; по нажатию снова переводить в `Loading` и запускать запрос в фоне (с тем же таймаутом).
- **Fallback:** при недоступности сервера можно показывать моковые данные (если включен офлайн-режим) или пустые списки с понятным сообщением («Не удалось загрузить каналы» и т.п.), но не блокировать UI.
- **Проверка:** отключить сеть или поднять «зависший» мок-сервер и убедиться, что интерфейс остаётся отзывчивым, через 10–15 с появляется ошибка/таймаут, есть возможность повторить загрузку.

### 8.3 Где применять

- Загрузка списка каналов при входе на сервер.
- Загрузка участников сервера/канала.
- Загрузка сообщений при выборе канала.
- Любые другие запросы, инициируемые из UI (настройки, инвайты и т.д.).

## 9. СИНХРОНИЗАЦИЯ В РЕАЛЬНОМ ВРЕМЕНИ (ИЗМЕНЕНИЯ С СЕРВЕРА)

При любом изменении на сервере все подключённые клиенты должны видеть это **сразу**, без перезагрузки и без ручного обновления.

### 9.1 Требование

Любые изменения на сервере должны **мгновенно** отражаться в UI у всех клиентов, в том числе:

- **Серверы:** добавление пользователя на сервер, удаление с сервера, смена названия/иконки сервера, список серверов пользователя.
- **Каналы:** создание, удаление, переименование, смена категории, смена порядка.
- **Участники и профили:** смена ника, аватарки, статуса, ролей; появление/исчезновение участника в списке (присоединился/вышел).
- **Сообщения:** новые сообщения, редактирование, удаление, реакции (если есть).
- **Прочее:** права, инвайты, закреплённые сообщения и т.д. — по мере появления фич на сервере.

Никакого «обнови страницу» или «зайди заново»; источник правды — сервер, доставка — по WebSocket (или аналог).

### 9.2 Реализация

- **Канал доставки:** использовать существующий WebSocket и очередь событий (`ws_events`, `process_ws_events` в `ui.rs`). Сервер при любом изменении шлёт событие подписанным клиентам; клиент в каждом кадре обрабатывает накопившиеся события и обновляет state.
- **Типы событий:** для каждой сущности — отдельный тип события (например `server.updated`, `channel.created`, `channel.renamed`, `channel.deleted`, `member.updated`, `user.updated` для ников/аватарок и т.д.). В плане и коде явно перечислить, какие события обрабатываются; при добавлении новой фичи на бэкенде — добавить событие и обработчик.
- **Обработка в UI:** в `process_ws_events` (или аналоге в модульной структуре) по типу события обновлять соответствующие данные в state: списки серверов, каналов, участников, кэш пользователей (ник, аватар), сообщения. После обновления — `ctx.request_repaint()`, чтобы UI перерисовался в следующем кадре.
- **Идемпотентность и дедупликация:** одно и то же событие может прийти один раз; обновление по id (заменить элемент в списке или в кэше, добавить новый, удалить по id). Для сообщений уже есть дедупликация по `message.id`; аналогично для каналов/участников по их id.
- **Аватары и медиа:** при событии смены аватарки пользователя — инвалидировать кэш аватара по `user_id` (удалить из `avatar_textures` / пометить устаревшим) и при необходимости поставить в очередь повторную загрузку; следующий показ этого пользователя подтянет новое изображение.

### 9.3 Чеклист событий (должны быть на сервере и обработаны на клиенте)

| Что меняется | Событие (пример) | Действие на клиенте |
|--------------|-------------------|----------------------|
| Список серверов пользователя | `server.list_updated` или отдельные add/remove | Обновить `servers` в state |
| Данные сервера (название, иконка) | `server.updated` | Обновить запись в списке серверов |
| Новый канал | `channel.created` | Добавить в `channels` для данного server_id |
| Канал переименован/изменён | `channel.renamed` / `channel.updated` | Обновить запись в списке каналов |
| Канал удалён | `channel.deleted` | Удалить из списка; если открыт — переключить вид |
| Участник присоединился/роль изменилась | `member.added` / `member.updated` | Обновить список участников |
| Ник/аватар пользователя | `user.updated` или `member.updated` | Обновить кэш пользователя и инвалидировать аватар |
| Новое сообщение | `message.created` | Уже есть; добавить в список сообщений канала |
| Сообщение отредактировано/удалено | `message.updated` / `message.deleted` | Обновить или удалить в списке сообщений |

Список расширять по мере появления новых сущностей на сервере.

**Реализовано (этап 12):** Клиент обрабатывает: `message.created`, `message.updated`, `message.deleted`, `channel.created`, `channel.renamed`, `channel.updated`, `channel.deleted`, `server.deleted`, `server.updated`, `server.added`, `member.joined`, `member.left`, `member.renamed`, `member.updated`, `user.updated`, `presence.init`, `presence.update`, `typing`, `messages.read`, `voice.*`. Сервер рассылает: все перечисленные, где есть соответствующие API (напр. `user.updated` при SetAvatar).

### 9.4 Поведение при расхождении

- Если клиент получил событие об удалении канала, в котором пользователь сейчас находится — переключить на первый доступный канал или на «общий» вид сервера и показать уведомление при необходимости.
- При обновлении текущего пользователя (ник/аватар) — обновить отображение в блоке профиля и в нижней панели без мерцания (данные подставляются из state).

## РЕКОМЕНДАЦИИ ПОСЛЕ ОСНОВНОГО UI

### LiveKit voice UI

- Вынести голосовой UI в отдельный слой/панель: индикаторы участников (кто говорит), кнопки Mute/Deafen, список участников в звонке.
- Использовать уже существующий `voice_livekit` и при необходимости `voice_engine_tx`/состояние голоса из текущего `ui.rs`.
- В `member_panel` и `bottom_panel` подписываться на состояние «speaking» и «muted» из LiveKit и отображать их через Theme (например, accent для speaking).

### Drag-and-drop серверов

- В egui нет нативного DnD для произвольных виджетов; возможные варианты:
  - Использовать `egui::LayerId` и проверять `ctx.input().pointer.drag_started()` / `pointer_pos()` над иконкой сервера, затем рисовать «призрак» и определять drop-зону между двумя иконками.
  - Либо подключить крейт с DnD (если найдётся совместимый с egui 0.28) и привязать к порядку серверов в state (массив или список id с сохранением порядка в настройках).

### Кастомная тема Astrix

- В `theme.rs` хранить не только цвета, но и опциональные параметры (радиус скругления, размеры отступов).
- Добавить в настройки (например в `Settings`) выбор темы: `ThemePreset::DiscordDark`, `ThemePreset::DiscordLight`, `ThemePreset::Astrix` (кастомная).
- Для Astrix задать отдельную палитру и, при необходимости, отличия в контрасте/акцентах; загрузка темы из конфига/файла — расширение на будущее.
